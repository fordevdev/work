# Multi-hop Agentic Retrieval

일반적인 [RAG](/notes/concepts/rag)는 "검색 한 번 → 생성 한 번"으로 끝난다. 하지만 여러 소스를 엮어야 답이 나오는 질문("이 기능을 왜 이렇게 설계했나요?" 같은, JIRA 티켓 + Confluence 설계문서 + Mattermost 논의 스레드가 다 필요한 질문)은 한 번의 검색만으로는 관련 정보를 다 못 찾는다.

## 동작 방식

1. 질의로 1차 검색
2. 결과를 보고 "이걸로 충분한가, 아니면 더 찾아야 하나"를 LLM이 스스로 판단
3. 부족하면 [graph](/notes/concepts/knowledge-graph)의 연결 문서를 따라가거나, 중간 결론을 바탕으로 새로운 질의를 만들어 재검색
4. 충분한 근거가 모이면 최종 답변 생성

즉, "검색 → 중간 추론 → 재검색"을 필요한 만큼 반복하는 loop다. Agent가 스스로 다음 행동(더 찾을지, 답할지)을 결정한다는 의미에서 "agentic"이라고 부른다.

## 트레이드오프

- **장점**: 단발 검색으로는 놓치는 복잡한 질문에 정확하게 답할 수 있다.
- **단점**: 검색을 여러 번 반복하므로 응답 지연(latency)과 LLM 호출 비용이 늘어난다.

그래서 이 프로젝트에서는 모든 질문에 무조건 multi-hop을 쓰지 않고, 1차 검색 결과의 신뢰도가 낮거나 질의가 복잡하다고 판단될 때만 추가 hop을 타도록 설계한다.
