2. 컴포넌트 딥다이브

전체 6단계 중 설계·운영 난도가 가장 높다고 판단한 3곳. MCP 서버·봇 오케스트레이션은 정형화된 패턴이 이미 있어 상대적으로 공수가 낮습니다.

A. 수집·동기화 파이프라인

effort: high

API 3종의 인증·레이트리밋이 제각각이고, 증분 동기화에서 누락·중복을 막는 로직 자체가 오래 걸리는 지점입니다.

  • 커넥터 — JIRA/Confluence REST, Mattermost API (각기 다른 페이지네이션/레이트리밋)
  • 변경 감지 — 웹훅 우선, 미지원 구간은 폴링+diff
  • 청킹·임베딩 — 문서 구조(제목/코드블록) 보존한 청크 분할
  • 메타 태깅 — 출처·작성자·수정일·권한 스코프 부착
  • 업서트 — 동일 문서 재수집 시 중복 방지 키 관리

B. 하이브리드 검색·랭킹

effort: high

네 가지 신호의 가중치 튜닝과, 오래됐지만 여전히 맞는 문서를 억울하게 밀어내지 않는 균형이 핵심 난제입니다.

score = sim(q,d) × time_decay(d) × authority(d) × feedback(d)
  • 벡터 검색 — 질의 임베딩과 근접한 청크 후보군 확보
  • 그래프 확장 — supersedes / references 엣지로 관련 문서 추가 탐색
  • 충돌 해소 — 대체된 문서는 자동 하위 랭크 또는 제외
  • 멀티홉 — 복잡 질의는 검색→중간추론→재검색 에이전틱 루프

C. 피드백 어텐션·가중치 갱신

effort: medium-high

답변 하나가 여러 청크를 조합하기 때문에, 반응 하나를 어느 청크의 공로/과실로 돌릴지가 근본적으로 모호합니다.

  • 수집 — 봇 답변에 대한 이모지 리액션·스레드 응답
  • 어텐션 매핑 — 답변 생성에 실제 인용된 청크에만 귀속
  • 스무딩 — (👍+1)/(전체+2) 베이지안 추정으로 소수 피드백 과민반응 방지
  • 감쇠·재계산 — 오래된 피드백 비중 축소, 주기적 배치 재계산
  • 유저 다양성 보정 — 동일인 반복 반응 디스카운트

참고: MCP 서버(툴 스펙 정의)와 Mattermost 봇(멘션 수신·답변 게시)은 각각 표준 MCP 서버 보일러플레이트와 Mattermost Bot API로 구현 가능해 상대적으로 예측 가능한 공수입니다 — 초기 스프린트는 A·B·C에 집중하는 것을 권장합니다.
